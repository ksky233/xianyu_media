# Dashboard 功能规划

## 当前结论

Dashboard 功能暂时不进入实现阶段。当前项目的业务板块还比较集中，核心可用功能主要是视频资源管理，此时直接增加 Dashboard 容易变成单一视频模块的统计页，信息密度和使用价值都不够稳定。更合适的时机是后续项目出现更多业务模块之后，例如视频、相册、任务、素材、账号、发布记录等模块逐渐完整，再把 Dashboard 作为整个后台的入口工作台来设计。

当前阶段只保留功能规划，不新增前后端代码，不新增数据库表，也不做接口迁移。

## 功能定位

Dashboard 不应做成装饰性大屏，而应作为个人后台工具的工作台。它打开后应该快速回答几个问题：当前资源库整体状态如何，哪些数据需要处理，最近有什么新增内容，哪些资源可能存在问题，下一步最值得操作的事项是什么。

第一版 Dashboard 可以定位为“资源管理总览”，重点放在可操作的信息上，而不是图表数量。后续如果项目模块增加，再扩展成跨模块总览。

## 第一版建议板块

后续实现时，第一版可以包含以下内容：

```txt
顶部统计卡片
  - 总视频数
  - 有效资源数
  - 失效或缺失资源数
  - 最近 7 天新增数

资源状态概览
  - 按 url_status 聚合统计
  - 展示主流网盘有效、其他网盘有效、全部失效等状态

类型分布
  - 按 type 聚合统计
  - 展示电影、电视剧、动漫、短剧、其他等分类

数据待补全
  - 缺封面数量
  - 缺简介数量
  - 缺年份数量
  - 缺网盘链接数量

最近新增
  - 最近 5 条视频
  - 点击后进入视频编辑页
```

这些板块都可以直接基于 `videos` 表聚合出来，第一版不需要新增 Dashboard 专属数据表。

## 后端规划

后端按业务模块化方式新增 `dashboard` 模块。虽然第一版主要是聚合查询，仍然建议保留完整分层结构，因为规范比省事重要，后续 Dashboard 一旦扩展更多统计项、跨模块聚合或复杂查询，目录结构不需要再返工。

建议结构如下：

```txt
xianyu_backend/app/modules/dashboard/
  __init__.py
  router.py
  deps.py
  service.py
  crud.py
  schema.py
```

各文件职责如下：

```txt
router.py
  定义 Dashboard HTTP 接口，例如 GET /api/v1/dashboard/overview。

deps.py
  定义 Dashboard 模块内部依赖，例如 DashboardCRUD、DashboardService 的依赖注入。

service.py
  组织业务逻辑，组合多个统计结果，决定最终返回结构。

crud.py
  执行数据库查询，包括 count、group by、最近新增列表、缺失字段统计等。

schema.py
  定义 Dashboard 接口的请求和响应模型。
```

第一版接口建议只做一个：

```txt
GET /api/v1/dashboard/overview
```

该接口需要登录认证，和视频管理接口保持一致。后端返回结构可以先设计为一个总览对象：

```json
{
  "code": 1,
  "msg": "操作成功",
  "data": {
    "summary": {
      "total_videos": 120,
      "valid_links": 93,
      "invalid_links": 12,
      "created_last_7_days": 8
    },
    "url_status_stats": [
      { "status": 0, "label": "全部失效", "count": 12 },
      { "status": 1, "label": "主流网盘有效", "count": 70 }
    ],
    "type_stats": [
      { "type": "电影", "count": 30 },
      { "type": "电视剧", "count": 25 }
    ],
    "incomplete_stats": {
      "missing_cover": 10,
      "missing_description": 18,
      "missing_year": 6,
      "missing_cloud_url": 12
    },
    "recent_videos": []
  }
}
```

后端路由注册位置：

```py
api_router.include_router(dashboard_router, prefix="/dashboard", tags=["dashboard"])
```

## 前端规划

前端按当前 feature 结构新增 `dashboard` 业务模块，保持和 `video` 模块一致的组织方式。

建议结构如下：

```txt
xianyu_frontend/src/features/dashboard/
  api/dashboard-api.ts
  hooks/use-dashboard-overview.ts
  pages/DashboardPage.tsx
  schemas/dashboard-schema.ts
  sections/DashboardSummarySection.tsx
  sections/DashboardResourceStatusSection.tsx
  sections/DashboardIncompleteSection.tsx
  sections/DashboardRecentVideosSection.tsx
```

请求链路保持当前项目统一模式：

```txt
DashboardPage
  -> useDashboardOverview()
    -> useQuery()
      -> getDashboardOverview()
        -> apiClient('/api/v1/dashboard/overview')
```

路由规划：

```txt
/dashboard -> DashboardPage
/          -> Navigate to /dashboard
```

侧边栏增加 Dashboard 菜单项，视频管理继续保留为独立菜单项。

## 是否需要新增依赖

第一版不建议新增图表库。Dashboard 可以先使用统计卡片、列表、横向进度条和简单分组布局完成。这样实现成本低，也更符合当前项目的个人后台工具定位。

如果后续确实需要趋势图、柱状图、饼图，再考虑引入 `recharts`。图表库应在数据结构稳定后再加，避免早期为了展示效果引入额外复杂度。

## 实现顺序

后续真正开始实现时，建议按以下顺序推进：

```txt
1. 明确 Dashboard 第一版展示板块
2. 定义后端 schema.py 响应结构
3. 实现 dashboard/crud.py 聚合查询
4. 实现 dashboard/service.py 组合业务数据
5. 实现 dashboard/router.py 接口
6. 在 modules/router.py 注册 Dashboard 路由
7. 前端定义 dashboard-schema.ts
8. 前端实现 dashboard-api.ts 和 use-dashboard-overview.ts
9. 实现 DashboardPage 与各个 section
10. 接入前端路由和侧边栏
11. 跑后端接口验证、前端 build/lint
```

## 暂缓原因

当前暂缓 Dashboard 的原因不是技术上不能做，而是功能收益还不够高。现在项目核心模块仍然集中在视频管理，Dashboard 很容易只重复展示视频列表已有信息。等后续模块增加后，Dashboard 可以自然承担“跨模块汇总”和“工作入口”的角色，那时它的价值会更清楚，结构也会更稳定。
