# 项目组织方式

## 两种常见组织思路

前后端项目中常见的代码组织方式大体可以分成两类：一种是按技术职责横向分层，另一种是按业务功能纵向聚合。横向分层更像传统 MVC 或 Java 后端项目，代码按照 `controller`、`service`、`repository`、`model`、`schema` 等技术角色分目录；纵向聚合更像 `features` 或 `modules` 架构，代码按照 `video`、`auth`、`user`、`album` 等业务模块分目录。

这两种组织方式都可以写出正确代码，但它们服务的思维不同。横向分层强调每层职责清楚，适合快速理解技术分工；纵向聚合强调业务边界清楚，适合围绕一个功能集中开发和维护。

## 横向分层

横向分层按照技术类型组织代码，典型后端结构如下：

```text
app/
  api/
  service/
  crud/
  models/
  schemas/
```

典型前端结构如下：

```text
src/
  pages/
  components/
  api/
  stores/
  schemas/
  hooks/
```

这种结构的优点是职责直观。看到 `api` 就知道是接口层，看到 `service` 就知道是业务层，看到 `schemas` 就知道是数据结构和校验层。它对初学者友好，也符合很多传统框架和教程的讲解方式。

它的问题是业务变复杂后，同一个功能会散落在多个目录。比如“视频管理”可能同时涉及页面、表格组件、详情弹窗、接口请求、表单 schema、状态管理、后端 router、service、repository 和 model。开发一个视频功能时，需要在很多横向目录之间来回跳转。

## 按业务模块聚合

按业务模块聚合会把同一个业务功能相关的代码放在一起。前端常叫 `features` 架构，后端常叫 `modules`、领域模块或按业务分包。

前端可以组织为：

```text
src/
  features/
    video/
      pages/
      sections/
      parts/
      api/
      schemas/
      hooks/

    auth/
      pages/
      api/
      schemas/

  components/
    ui/
    common/

  layouts/
  lib/
```

后端可以组织为：

```text
app/
  modules/
    video/
      router.py
      service.py
      repository.py
      schemas.py
      models.py

    auth/
      router.py
      service.py
      repository.py
      schemas.py

    user/
      router.py
      service.py
      repository.py
      schemas.py

  core/
  db/
  shared/
  main.py
```

这种结构的优点是业务聚合度高。开发视频模块时，视频相关的页面、组件、接口、schema、业务逻辑和数据库交互都在 `video` 模块附近，维护时更像在处理一个完整业务单元，而不是在技术分层之间来回移动。

## 推荐原则

本项目从现在开始推荐采用“按业务模块聚合为主，公共能力单独沉淀”的方式组织前后端代码。

核心原则是：

```text
业务相关代码就近放在业务模块内
跨业务复用代码放到 common / shared
纯 UI 基础组件放到 components/ui
框架级配置和基础设施放到 core / lib / db
先局部聚合，复用后上提
```

这套原则可以同时应用于前端和后端。前端里的 `features/video` 对应后端里的 `modules/video`，它们都表达“视频业务是一组完整能力，而不是散落在各个技术目录中的文件集合”。

## 前端推荐结构

对于当前 Vite React 前端，推荐从一开始就使用轻量业务模块结构：

```text
src/
  app/
    router.tsx
    providers.tsx

  layouts/
    MainLayout.tsx

  features/
    video/
      pages/
        VideoListPage.tsx
        VideoNewPage.tsx
        VideoEditPage.tsx

      sections/
        VideoSearchSection.tsx
        VideoTableSection.tsx
        VideoPaginationSection.tsx

      parts/
        VideoStatusBadge.tsx
        VideoDetailModal.tsx
        VideoForm.tsx

      api/
        video-api.ts

      schemas/
        video-schema.ts

      hooks/
        use-video-list.ts

    auth/
      pages/
        LoginPage.tsx

      api/
        auth-api.ts

      schemas/
        auth-schema.ts

      stores/
        auth-store.ts

    album/
      pages/
        AlbumPage.tsx

  components/
    ui/
      Button.tsx
      Input.tsx

    common/
      EmptyState.tsx
      LoadingState.tsx

  lib/
    api-client.ts
    utils.ts
```

这里的 `features` 负责业务模块，`components/ui` 负责纯 UI 基础组件，`components/common` 负责跨业务的通用组件，`layouts` 负责页面骨架，`lib` 负责和业务无关的工具与基础设施。

页面内部仍然保留熟悉的 `Page / Section / Part` 层级。`Page` 是路由级页面，`Section` 是页面大区块，`Part` 是业务页面内部的小零件，例如状态标签、详情弹窗、表单和卡片。这里使用 `parts` 而不是 `components`，是为了避免和全局 `components` 目录语义冲突。

## 后端推荐结构

对于当前 FastAPI 后端，推荐从传统横向结构逐步演进到业务模块结构：

```text
app/
  modules/
    video/
      router.py
      service.py
      repository.py
      schemas.py
      models.py
      workflow/
        recognizer.py
        cover_generator.py

    auth/
      router.py
      service.py
      repository.py
      schemas.py

    user/
      router.py
      service.py
      repository.py
      schemas.py

  core/
    config.py
    security.py
    logger.py
    exception.py

  db/
    session.py
    base.py

  shared/
    pagination.py
    response.py

  main.py
```

其中 `router.py` 对应 API 层，`service.py` 对应业务逻辑层，`repository.py` 对应数据库交互层，`schemas.py` 对应请求和响应结构，`models.py` 对应 ORM 模型。它们仍然保留清晰的分层职责，只是这些职责被放到同一个业务模块中。

这意味着业务分层没有消失，而是从横向全局目录变成了模块内部的小分层。比如 `modules/video/service.py` 仍然是 service，只是它和视频相关的 router、repository、schema 放在一起。

## 当前项目如何迁移

前端目前处于新 Vite 项目阶段，可以从一开始就采用推荐结构。也就是说，后续迁移登录、视频和专辑页面时，优先放入 `features/auth`、`features/video` 和 `features/album`，不要继续把所有页面堆在全局 `pages` 下，也不要把所有业务组件堆在全局 `components` 下。

后端目前已经有较完整的横向结构：

```text
app/
  api/
  service/
  crud/
  models/
  schemas/
```

后端可以暂时不立刻大规模移动文件，但新的业务能力和后续重构应当向 `modules` 结构靠拢。比如视频模块最适合优先迁移，因为它已经包含 API、CRUD、service、schema、model 和 workflow。可以先从 `video` 模块开始，把相关文件聚合到：

```text
app/modules/video/
```

等视频模块稳定后，再考虑迁移 `auth` 和 `user`。这样可以避免一次性重构过大，同时让后端逐步获得业务模块化结构。

## 小项目是否需要这种结构

小项目同样适合采用业务模块聚合方式，尤其是学习型项目。原因是小项目改动成本低、上下文简单、试错空间大，适合从一开始练习良好的工程组织习惯。

如果等项目变大后再迁移，文件引用、业务耦合、测试、文档和部署路径都会增加迁移成本。小项目阶段提前建立清晰边界，可以让后续功能增长时自然落到已有结构中。

因此，本项目不把业务模块化视为大型项目专属实践，而是把它作为长期工程习惯来使用。

## 最终推荐

本项目推荐统一采用以下方向：

```text
前端：features 按业务聚合，components 只放跨业务复用组件
后端：modules 按业务聚合，模块内部保留 router / service / repository / schemas / models 分层
公共能力：前端放 lib / components/common，后端放 core / db / shared
迁移节奏：前端新项目直接采用，后端从 video 模块开始逐步迁移
```

一句话总结：横向分层适合说明技术职责，业务模块聚合适合长期维护；本项目从现在开始以后者为主，并在模块内部保留必要的技术分层。
