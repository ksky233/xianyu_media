# 按业务模块划分后端项目

## 当前项目适合按业务拆分

当前后端虽然目录形态接近传统 MVC，但业务边界已经比较清楚，视频、用户、Token 这几类业务基本都遵循了 `api -> service -> crud -> model/schema` 的依赖方向。这样的项目迁移到按业务模块划分时，核心工作不是重写业务逻辑，而是把同一个业务散落在不同技术层目录里的文件收拢到同一个业务目录里。

例如视频业务当前分散在 `api/v1/video.py`、`service/video_service.py`、`crud/crud_video.py`、`models/video.py`、`schemas/video_schema.py`，按业务模块划分后可以收拢为 `modules/videos/router.py`、`modules/videos/service.py`、`modules/videos/crud.py`、`modules/videos/model.py`、`modules/videos/schema.py`。用户业务和 Token 业务也可以用同样方式处理。

## 当前项目迁移的主要风险点

迁移风险主要来自公共基础设施和框架装配层，而不是来自业务文件本身。业务文件移动后，只要导入路径修正，原有逻辑通常可以继续工作；但数据库、依赖注入、路由注册、Alembic 模型发现这些位置一旦处理不完整，就会出现服务启动失败、接口依赖缺失、迁移脚本识别不到表等问题。

当前最需要小心的是 `app/api/deps.py`。这个文件现在把数据库 session、API Token 验证、JWT 用户验证、用户服务注入、视频服务注入都放在一起。按业务拆分后，如果继续保留一个巨大的全局 `deps.py`，目录看起来已经模块化了，实际依赖仍然集中在一个文件里，后续每新增一个业务模块都要继续修改这个公共文件，模块之间的边界会重新变模糊。

更合适的处理方式是把真正公共的依赖保留在 shared 层，例如数据库 session、基础认证工具、当前用户解析等；把业务专属的依赖放回业务模块内部，例如 `modules/videos/deps.py` 只负责构造 `VideoCRUD` 和 `VideoService`，`modules/users/deps.py` 只负责构造 `UserCRUD` 和 `UserService`。这样 API 层只从当前业务模块或 shared 层导入依赖，不需要跨过业务边界去引用一堆不相关对象。

另一个风险点是 `app/models/__init__.py` 和 Alembic。当前 `app/models/__init__.py` 同时承担数据库 engine、session factory、Base 导出、模型导入这几件事，而 Alembic 的 `env.py` 又通过 `from app.models import Base` 获取 `Base.metadata`。如果把模型移动到 `modules/videos/model.py`、`modules/users/model.py` 以后，没有在统一位置显式导入这些模型，Alembic 可能拿到一个空的或不完整的 metadata，自动生成迁移时就识别不到表结构变化。

未来更清晰的做法是把数据库连接能力放到 `shared/db.py`，例如 `engine`、`AsyncSessionFactory`、`Base` 都从这里导出；再提供一个明确的模型注册入口，例如 `app/modules/__init__.py` 或 `app/shared/model_registry.py`，专门导入所有业务模块的 SQLAlchemy model。Alembic 只依赖这个统一入口，服务启动也只依赖这个统一入口，这样模型文件可以按业务分散，元数据注册仍然集中且可控。

## 推荐的目标结构

一个适合当前项目的后端结构可以是：

```txt
app/
  main.py

  shared/
    db.py
    deps.py
    response.py
    exception.py
    logger.py
    config.py
    security/
      jwt.py

  modules/
    videos/
      router.py
      deps.py
      service.py
      crud.py
      model.py
      schema.py

    users/
      router.py
      deps.py
      service.py
      crud.py
      model.py
      schema.py

    api_tokens/
      router.py
      deps.py
      service.py
      crud.py
      model.py
      schema.py

  workflows/
    video_recognizer.py
    cover_generator.py
```

这里的 `modules` 表示业务模块，`shared` 表示跨业务复用的基础设施。一个判断标准是：如果某个文件必须知道某个具体业务，比如 `VideoService`、`VideoCRUD`、`VideoCreate`，它就应该靠近视频模块；如果某个文件不关心具体业务，比如数据库 session、统一响应、异常类型、日志、配置、JWT 编解码，它就应该放在 shared 层。

## 容易堆在一起的模块如何处理

按业务拆分后，最容易重新堆在一起的是依赖注入、数据库模型注册、统一响应、异常处理、权限认证、工具函数。它们看起来都“到处会用”，所以很容易被放进一个越来越大的 `common` 或 `deps` 文件里。更稳定的做法是先区分“技术公共能力”和“业务公共能力”。

技术公共能力包括数据库连接、配置、日志、异常基类、统一响应、JWT 编解码，这些可以放进 `shared`。业务公共能力需要更谨慎，例如“当前用户必须是管理员”虽然很多接口都会用，但它仍然属于用户认证上下文，可以放在 `modules/users/deps.py` 或 `shared/security/deps.py`，并保持它只暴露最小接口，不反向依赖视频、专辑等业务。

数据库模型注册也要单独对待。模型文件可以放在各自业务模块里，但 `Base.metadata` 的完整性必须由一个明确入口保证。也就是说，模型“物理位置”可以分散，模型“注册入口”要集中。这样既能按业务维护模型，又能让 Alembic 和 SQLAlchemy 稳定发现全部表结构。

路由注册也是类似思路。每个业务模块自己暴露 `router`，`main.py` 或一个 `modules/router.py` 负责统一 include。这样主应用只关心“挂载哪些模块”，不关心每个模块内部的 service、crud、schema 如何组织。

## 当前项目的判断

当前项目适合迁移到按业务模块划分，而且迁移收益比较明显。视频业务已经形成完整闭环，用户和 Token 也有清晰边界；真正要提前设计好的，是全局依赖注入如何拆、数据库模型如何注册、Alembic 如何继续拿到完整 metadata。只要这两个基础点处理稳，后面的业务模块迁移就主要是移动文件和修正导入路径。
