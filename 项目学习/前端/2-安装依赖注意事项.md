# 安装依赖注意事项

## 安装依赖前先确认职责

安装依赖时不要只根据包名相似就直接安装，因为前端生态里很多包看起来属于同一技术栈，实际承担的职责完全不同。正确的做法是先确认这个包解决什么问题、属于运行时依赖还是开发依赖、是否需要配套插件，以及当前项目是否已经进入需要它的阶段。

以 Tailwind 为例，`tailwindcss`、`@tailwindcss/vite` 和 `tailwind-merge` 都和 Tailwind 有关，但它们不是同一个东西的不同版本，而是不同职责的包。

## tailwindcss 是样式框架本体

`tailwindcss` 是 Tailwind CSS 的核心包，它负责提供工具类体系，并在构建过程中把项目中使用到的 class 转换成最终 CSS。

例如页面中写：

```tsx
<div className="flex rounded-md bg-slate-950 px-4 py-2 text-white" />
```

这些 class 能够变成真实样式，依赖的是 Tailwind CSS 本体。

在 Vite 项目中安装 Tailwind 本体通常使用开发依赖：

```powershell
pnpm add -D tailwindcss
```

这里的 `-D` 表示开发依赖，因为 Tailwind 主要参与开发和构建过程，最终浏览器加载的是构建后的 CSS。

## @tailwindcss/vite 是 Vite 集成插件

`@tailwindcss/vite` 是 Tailwind 面向 Vite 的集成插件，它负责把 Tailwind 接入 Vite 的开发服务器和构建流程。

在 Tailwind v4 的 Vite 项目中，推荐同时安装：

```powershell
pnpm add -D tailwindcss @tailwindcss/vite
```

`@tailwindcss/vite` 这种包名中的 `@tailwindcss/` 是命名空间，表示这个包属于 Tailwind CSS 相关组织或包作用域，`vite` 才是具体包名。类似的命名方式还有 `@vitejs/plugin-react`、`@types/react`、`@tanstack/react-query`。

安装这个插件之后，还需要在 `vite.config.ts` 中接入它。依赖安装只是第一步，真正让它生效还需要按官方推荐完成配置。

## tailwind-merge 是 class 合并工具

`tailwind-merge` 不是 Tailwind 的核心包，也不是 Vite 插件，它是一个用于合并 Tailwind className 的工具库。

例如封装一个按钮组件时，组件默认样式里有：

```tsx
"px-4 py-2 bg-black"
```

调用组件时又传入：

```tsx
"px-2 bg-red-500"
```

普通字符串拼接会得到：

```tsx
"px-4 py-2 bg-black px-2 bg-red-500"
```

这里的 `px-4` 和 `px-2` 冲突，`bg-black` 和 `bg-red-500` 也冲突。`tailwind-merge` 的作用就是根据 Tailwind 的规则保留后面的有效类，得到更符合预期的结果。

因此它适合在开始封装 `Button`、`Input`、`Card` 等通用组件时安装，初始化 Tailwind 阶段可以先不安装。

## clsx 是条件拼接 className 的工具

`clsx` 常和 `tailwind-merge` 一起使用，它负责根据条件拼接 className。

例如：

```tsx
clsx("rounded-md", isActive && "bg-slate-950")
```

当 `isActive` 为真时，会拼接激活样式；当 `isActive` 为假时，会忽略这段样式。后续常见的工具函数是：

```ts
import { clsx, type ClassValue } from "clsx"
import { twMerge } from "tailwind-merge"

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```

这个函数适合在项目开始封装自己的 UI 组件时引入。当前只做基础依赖安装和技术栈学习时，可以先理解它的用途，再决定是否安装。

## 运行时依赖和开发依赖

安装依赖时还要区分运行时依赖和开发依赖。运行时依赖会被应用代码直接使用，例如 `react-router-dom`、`zustand`、`@tanstack/react-query`，这些通常使用：

```powershell
pnpm add 包名
```

开发依赖主要服务于构建、检查、转换和类型提示，例如 `tailwindcss`、`@tailwindcss/vite`、`typescript`、`vite`、`@types/react`，这些通常使用：

```powershell
pnpm add -D 包名
```

这不是绝对规则，但可以作为初步判断：如果浏览器运行的业务代码会直接 import 它，通常是运行时依赖；如果它主要服务于构建工具、代码检查、类型声明或编译流程，通常是开发依赖。

## 按阶段安装依赖

学习和迁移项目时，依赖最好按阶段安装，而不是一次性把所有可能会用到的包都装进来。

如果当前阶段只准备学习路由和 Tailwind，可以先安装：

```powershell
pnpm add react-router-dom
pnpm add -D tailwindcss @tailwindcss/vite
```

当开始迁移接口请求和后端数据缓存时，再安装：

```powershell
pnpm add @tanstack/react-query
```

当开始处理登录态、用户信息、侧边栏状态等客户端全局状态时，再安装：

```powershell
pnpm add zustand
```

当开始迁移登录表单、视频新增表单、视频编辑表单时，再安装：

```powershell
pnpm add zod react-hook-form @hookform/resolvers
```

当开始封装 UI 组件或需要图标时，再安装：

```powershell
pnpm add lucide-react clsx tailwind-merge class-variance-authority
```

这种方式可以让每个依赖都对应一个明确的学习目标和业务场景，避免项目在一开始就堆满暂时没有使用的包。

## 安装前先查推荐接入方式

安装某个技术栈相关依赖前，应先查看它在当前工具链下的推荐安装方式。不同版本、不同构建工具、不同框架下，同一个技术栈的安装步骤可能不同。

例如 Tailwind 在 Vite 项目中的推荐方式，除了安装 `tailwindcss` 本体，还需要安装 `@tailwindcss/vite` 插件，并在 `vite.config.ts` 中配置插件。这说明安装依赖时不能只凭包名执行 `pnpm add tailwindcss`，还要确认当前项目使用的是 Vite、Next.js、Webpack 还是其他工具链。

一个比较稳妥的安装习惯是：

```text
先确认当前项目工具链
再查看官方文档或主流模板
确认需要安装哪些包
区分运行时依赖和开发依赖
安装后查看 package.json 变化
运行 pnpm build / pnpm lint 验证项目健康
```

## 本项目当前建议

当前项目刚完成 Vite React TypeScript 初始化，仍处于模板阶段。下一步如果只学习和接入路由、Tailwind，推荐先安装最小必要依赖：

```powershell
pnpm add react-router-dom
pnpm add -D tailwindcss @tailwindcss/vite
```

安装后先不要急着继续堆依赖，应当先理解 `package.json` 的变化，再配置 `vite.config.ts` 和全局样式文件，最后运行：

```powershell
pnpm build
pnpm lint
```

只有当项目仍然能正常构建和检查通过时，再进入下一阶段依赖安装与业务迁移。
