# 清理模板并建立基础结构

## 本阶段目标

Vite 初始化完成之后，项目中仍然保留着官方模板的欢迎页、演示图片、计数器、文档链接和配套样式。它们证明了模板可以运行，但这些内容和闲鱼媒体后台没有业务关系。继续在模板欢迎页上直接叠加业务代码，会让示例代码、全局样式和真实业务混在一起，后续排查样式问题、组织目录结构和接入路由都会变得不清晰。

本阶段的目标，是把 Vite 默认欢迎页清理掉，建立一个干净的前端底座，并完成三个基础能力的接入：`@` 路径别名、Tailwind CSS、项目目录结构。完成之后，页面只保留一个极简首页，用来确认 Vite、React、TypeScript、Tailwind 和路径别名已经连通。

## 清理 Vite 默认欢迎页

当前模板中的 `src/App.tsx` 原本导入了 `useState`、`react.svg`、`vite.svg`、`hero.png` 和 `App.css`，页面中展示了 Vite 默认欢迎区、计数器、文档链接和社交链接。这些内容属于脚手架演示代码，项目进入正式迁移前应当清理。

清理后的 `src/App.tsx` 只保留应用入口职责：

```tsx
import { HomePage } from '@/pages/HomePage'

export default function App() {
  return <HomePage />
}
```

这段代码体现了一个重要变化：`App.tsx` 不再负责展示模板页面，而是作为应用根组件，转而渲染项目自己的首页组件。这里使用 `@/pages/HomePage` 导入页面组件，也顺便验证后续配置的 `@` 路径别名能够正常工作。

模板中的 `src/App.css` 被删除，因为模板页面消失后，`.hero`、`.counter`、`#center`、`#next-steps` 等样式都失去了对应组件。模板资源 `src/assets/hero.png`、`src/assets/react.svg`、`src/assets/vite.svg` 也被删除，因为当前项目已经不再引用这些图片。

## 建立 src 项目目录结构

清理模板后，需要让 `src` 目录从“演示页面结构”转向“业务项目结构”。当前建立的基础目录是：

```text
src/
  App.tsx
  index.css
  main.tsx
  components/
  layouts/
  lib/
  pages/
    HomePage.tsx
```

`components` 用于存放可复用组件，例如按钮、弹窗、表格、搜索栏和业务局部组件；`layouts` 用于存放页面布局，例如后台主布局、侧边栏布局和鉴权布局；`lib` 用于存放工具代码，例如 API client、状态管理、请求缓存、schema 和通用工具函数；`pages` 用于存放页面级组件，例如登录页、视频列表页、新增视频页和编辑视频页。

空目录中放置 `.gitkeep` 文件，是为了让后续初始化 Git 后这些目录可以被版本控制记录。Git 默认不会追踪空目录，`.gitkeep` 只是一个占位文件，用来表达“这个目录结构是项目设计的一部分”。

## 配置 @ 路径别名

项目逐渐变大后，如果一直使用相对路径导入代码，很容易出现多层级路径：

```tsx
import { HomePage } from '../../pages/HomePage'
```

路径别名可以让导入路径从项目源码根目录开始表达：

```tsx
import { HomePage } from '@/pages/HomePage'
```

这让文件移动、目录加深和模块整理都更稳定。Vite 项目中配置别名需要同时处理两个层面：Vite 构建层和 TypeScript 类型检查层。

`vite.config.ts` 中加入 Vite 运行时别名：

```ts
import { fileURLToPath, URL } from 'node:url'
import tailwindcss from '@tailwindcss/vite'
import react from '@vitejs/plugin-react'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [react(), tailwindcss()],
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url)),
    },
  },
})
```

这里的 `fileURLToPath(new URL('./src', import.meta.url))` 会把当前配置文件所在位置下的 `src` 目录转换成绝对文件路径。Vite 在开发服务器和生产构建时，会根据这个配置理解 `@/pages/HomePage` 这种路径。

`tsconfig.app.json` 中加入 TypeScript 路径映射：

```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

这让 TypeScript、编辑器提示和类型检查也能理解 `@` 指向 `src`。如果只配置 Vite，运行和构建可能能识别别名，但编辑器和 TypeScript 检查会报找不到模块；如果只配置 TypeScript，编辑器能识别路径，但 Vite 构建阶段无法解析别名。因此这两个配置需要成对出现。

## 接入 Tailwind CSS

当前项目已经安装了 `tailwindcss` 和 `@tailwindcss/vite`。Tailwind v4 在 Vite 项目中推荐通过 Vite 插件接入，因此 `vite.config.ts` 中需要把 Tailwind 插件加入 plugins：

```ts
plugins: [react(), tailwindcss()]
```

这一步让 Vite 的开发服务器和生产构建流程能够处理 Tailwind。安装依赖只是把包放进项目，配置插件才是让 Vite 真正接入 Tailwind。

接着在 `src/index.css` 中加入 Tailwind 入口：

```css
@import "tailwindcss";
```

当前清理后的 `index.css` 还保留了少量全局基础样式：

```css
:root {
  color: #172033;
  background: #f6f7fb;
  font-family:
    Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI",
    sans-serif;
  font-synthesis: none;
  text-rendering: optimizeLegibility;
  -webkit-font-smoothing: antialiased;
  -moz-osx-font-smoothing: grayscale;
}

* {
  box-sizing: border-box;
}

body {
  margin: 0;
  min-width: 320px;
  min-height: 100vh;
}

button,
input,
textarea,
select {
  font: inherit;
}

#root {
  min-height: 100vh;
}
```

这些样式属于全局底座：`:root` 设置基本字体和文字渲染效果，`*` 统一盒模型，`body` 清除默认外边距并保证页面最小高度，表单控件继承字体，`#root` 保证 React 根节点可以撑满视口。模板中大量和欢迎页相关的颜色变量、暗色主题、标题字号、计数器样式和页面边框都被移除，因为它们服务于模板演示页，而不是本项目业务。

## 做一个极简首页

为了验证 Tailwind、路径别名和基础页面结构已经生效，本阶段新增了 `src/pages/HomePage.tsx`：

```tsx
export function HomePage() {
  return (
    <main className="flex min-h-screen items-center justify-center bg-slate-100 px-6">
      <section className="w-full max-w-2xl rounded-lg bg-white p-8 shadow-sm">
        <p className="text-sm font-medium text-slate-500">Xianyu Media</p>
        <h1 className="mt-3 text-3xl font-semibold text-slate-950">前端迁移底座已就绪</h1>
        <p className="mt-4 leading-7 text-slate-600">
          当前页面用于确认 Vite、React、TypeScript、Tailwind CSS 和路径别名已经接通。下一步可以在这个干净结构上继续接入路由和业务页面。
        </p>
      </section>
    </main>
  )
}
```

这个页面的作用不是开始迁移业务，而是作为“底座探针”。它使用了 Tailwind 的布局类、颜色类、间距类、阴影类和字体类，能够直观确认 Tailwind 已经接入；它被 `App.tsx` 通过 `@/pages/HomePage` 导入，能够确认路径别名已经生效；它不依赖任何旧模板资源，能够确认默认欢迎页已经被彻底清理。

## 验证 build 和 lint

清理模板和接入基础配置之后，需要立即验证项目仍然健康。常规验证命令是：

```powershell
pnpm build
pnpm lint
```

`pnpm build` 会执行 `tsc -b && vite build`，因此它同时验证 TypeScript 类型检查和 Vite 生产构建；`pnpm lint` 会执行 `oxlint`，因此它验证当前代码是否符合 Oxlint 规则。

本阶段验证的重点是：

```text
@ 路径别名能被 TypeScript 和 Vite 同时识别
Tailwind 插件能参与 Vite 构建
index.css 中的 Tailwind 入口能正常处理
App.tsx 能正确渲染 HomePage
删除模板资源后没有残留引用
```

如果这两个命令都通过，说明当前前端底座已经从模板演示工程转换成了干净的项目工程，可以继续进入路由、状态管理、API 层和业务页面迁移。

## 本阶段完成后的状态

完成本阶段后，项目状态可以概括为：

```text
Vite 默认欢迎页已清理
模板专用 App.css 已删除
模板图片资源已删除
src 下建立 components、layouts、lib、pages 目录
App.tsx 渲染项目自己的 HomePage
vite.config.ts 接入 React 插件、Tailwind 插件和 @ 别名
tsconfig.app.json 接入 @ 路径映射
index.css 接入 Tailwind 并保留少量全局基础样式
build 和 lint 验证通过
```

这一步的价值在于把项目从“脚手架默认示例”整理为“业务项目起点”。后续迁移登录页、主布局、视频列表和接口层时，就可以落在清晰的目录结构中，而不是继续和模板演示代码纠缠在一起。
