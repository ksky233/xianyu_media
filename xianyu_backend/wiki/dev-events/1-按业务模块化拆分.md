# 按业务模块化拆分规划

## 背景

当前后端项目采用的是按技术层分目录的组织方式，也就是 `api`、`service`、`crud`、`models`、`schemas` 分别放在不同目录中。这个结构在项目早期非常直观，因为每一层职责都清楚；但当业务模块逐渐增加后，同一个业务会散落在多个目录里，例如视频业务同时存在于 `api/v1/video.py`、`service/video_service.py`、`crud/crud_video.py`、`models/video.py`、`schemas/video_schema.py`。

本次拆分的目标是把项目从“按技术层分目录”调整为“按业务模块分目录”。调整完成后，一个业务的路由、依赖、服务、数据库访问、模型、请求响应结构会尽量放在同一个业务目录中。这样做的价值是：后续维护某个业务时，开发者可以优先进入这个业务模块，而不是在全局目录中来回查找相关文件。

当前项目规模还不大，视频、用户、Token 三个主要业务边界也比较清楚，因此适合一次性完成模块化拆分。这里的一次性迁移指的是在一次重构周期中完成目录调整、导入路径修正、服务启动验证和接口验证，而不是边写新业务边慢慢兼容两套目录结构。

## 目标结构

目标结构建议如下：

```txt
app/
  main.py

  shared/
    config.py
    db.py
    logger.py
    response.py
    exception.py
    exception_handler.py
    decorator.py
    deps.py
    security/
      jwt.py

  modules/
    __init__.py
    router.py

    videos/
      __init__.py
      router.py
      deps.py
      service.py
      crud.py
      model.py
      schema.py

    users/
      __init__.py
      router.py
      deps.py
      service.py
      crud.py
      model.py
      schema.py

    api_tokens/
      __init__.py
      router.py
      deps.py
      service.py
      crud.py
      model.py
      schema.py

  workflows/
    __init__.py
    cover_generator.py
    mcp_tools.py
    video_recognizer.py
    video_recognizer_v2.py

  utils/
    llm_response_parser.py
```

其中 `modules` 表示业务模块，`shared` 表示跨业务复用的基础设施。判断一个文件应该放在哪里时，可以使用一个简单标准：如果文件必须理解某个具体业务概念，例如 Video、User、ApiToken，它就应该靠近业务模块；如果文件只提供通用技术能力，例如数据库 session、统一响应、日志、配置、异常处理、JWT 编解码，它就应该放到 `shared`。

## 业务边界说明

### videos

`videos` 是当前最完整的业务模块，包含视频资源的创建、查询、条件搜索、详情、更新、删除、批量删除。它应当拥有自己的路由、服务、CRUD、模型和 schema。

视频模块可以依赖 shared 层的数据库、异常、统一响应、日志，也可以依赖用户模块暴露出来的“当前管理员用户”依赖。但视频模块不应该直接理解用户密码、Token 创建、JWT 生成等认证细节。

### users

`users` 负责用户登录、用户资料、修改密码、当前用户解析、管理员权限判断。当前 `user.py` 路由中包含用户登录逻辑，`deps.py` 中也包含当前用户和管理员用户的依赖，这部分在迁移时需要收拢。

建议 `modules/users/deps.py` 提供：

```txt
get_user_crud
get_user_service
get_current_user
get_current_user_admin
```

这样其他业务如果需要用户身份，只导入 `get_current_user` 或 `get_current_user_admin`，不用直接接触 JWT 解析和用户查询细节。

### api_tokens

当前 `api/v1/auth.py` 实际做的是 API Token 管理和验证，不是用户登录。为了避免和用户登录混淆，业务模块建议命名为 `api_tokens`，路由前缀仍然可以保持 `/auth` 或调整为 `/api-tokens`。

如果为了减少前端和外部调用影响，本次可以保留原有路径：

```txt
/api/v1/auth/tokens
/api/v1/auth/tokens/verify
```

模块目录命名和 HTTP 路由命名不必须完全一致。目录名表达代码归属，路由名表达对外接口契约。

## 依赖注入拆分方案

当前 `app/api/deps.py` 是本次迁移最需要小心的文件。它同时包含：

```txt
数据库 session
API Token 的 HTTPBearer 提取
API Token 验证
UserCRUD / UserService 构造
JWT 当前用户解析
管理员权限判断
VideoCRUD / VideoService 构造
```

如果只是移动业务文件，但继续保留这个巨大的全局 deps 文件，那么项目表面上已经模块化，实际依赖仍然集中在一个地方。后续每新增一个业务模块，都要继续修改全局 `deps.py`，模块边界会再次被打穿。

推荐拆法如下：

```txt
app/shared/deps.py
  get_db_session

app/shared/security/deps.py
  get_bearer_token 或通用认证提取能力

app/modules/users/deps.py
  get_user_crud
  get_user_service
  get_current_user
  get_current_user_admin

app/modules/api_tokens/deps.py
  get_token_crud
  get_api_token_service
  verify_api_token

app/modules/videos/deps.py
  get_video_crud
  get_video_service
```

这里的关键是：公共依赖只保留真正公共的东西，业务依赖回到业务模块内部。视频路由需要视频服务时，从 `app.modules.videos.deps` 导入；视频路由需要管理员身份时，从 `app.modules.users.deps` 导入。这样依赖方向是明确的。

迁移后视频路由中的依赖关系应该类似：

```python
from app.modules.users.deps import get_current_user_admin
from app.modules.videos.deps import get_video_service
```

这种导入是可以接受的，因为视频接口确实依赖“管理员用户”这个权限能力。但视频模块不应该导入 `UserCRUD`、`UserService` 或 JWT 工具来自己判断用户身份。

## 数据库模型和 Alembic 处理方案

第二个高风险点是 SQLAlchemy model 注册和 Alembic metadata。

当前 `app/models/__init__.py` 同时负责：

```txt
创建 engine
创建 AsyncSessionFactory
导出 Base
导入 ApiToken / User / Video 模型
```

Alembic 当前通过以下方式获取 metadata：

```python
from app.models import Base
target_metadata = Base.metadata
```

如果把模型移动到业务模块后，没有显式导入所有模型，Alembic 可能拿到不完整的 `Base.metadata`。这种问题很隐蔽，因为服务可能可以启动，但自动生成迁移时识别不到某些表。

推荐处理方式是：

```txt
app/shared/db.py
  Base
  engine
  AsyncSessionFactory
  get_db_session

app/modules/__init__.py
  import app.modules.api_tokens.model
  import app.modules.users.model
  import app.modules.videos.model
```

Alembic 的 `env.py` 调整为：

```python
from app.shared.db import Base
import app.modules

target_metadata = Base.metadata
```

这里 `import app.modules` 的目的不是使用其中某个变量，而是触发所有业务模型模块被导入，从而完成 SQLAlchemy model 注册。模型文件可以按业务分散，但注册入口必须集中、明确、稳定。

如果希望语义更清楚，也可以创建：

```txt
app/shared/model_registry.py
```

里面专门导入所有模型：

```python
from app.modules.api_tokens.model import ApiToken
from app.modules.users.model import User
from app.modules.videos.model import Video
```

然后 Alembic 导入：

```python
from app.shared.db import Base
import app.shared.model_registry

target_metadata = Base.metadata
```

这两种方式都可以。更推荐 `model_registry.py`，因为它把“导入模型是为了注册 metadata”这个意图表达得更明确。

## 路由注册方案

当前 `main.py` 直接导入三个路由：

```python
from app.api.v1.auth import router as auth_router
from app.api.v1.user import router as user_router
from app.api.v1.video import router as video_router
```

迁移后可以改成：

```python
from app.modules.router import api_router
```

`app/modules/router.py` 负责聚合所有业务路由：

```python
from fastapi import APIRouter

from app.modules.api_tokens.router import router as api_token_router
from app.modules.users.router import router as user_router
from app.modules.videos.router import router as video_router

api_router = APIRouter()
api_router.include_router(api_token_router)
api_router.include_router(user_router)
api_router.include_router(video_router)
```

`main.py` 只负责挂载：

```python
app.include_router(api_router, prefix=settings.API_V1_STR)
```

这样 `main.py` 不需要知道具体业务模块越来越多的细节，只知道挂载 API 总路由。

## 导入路径规则

迁移后建议统一使用绝对导入，避免相对路径层级太深后变得难读。

推荐：

```python
from app.modules.videos.schema import VideoCreate
from app.modules.videos.service import VideoService
from app.shared.response import UnifiedResponse
```

不推荐：

```python
from .schema import VideoCreate
from ...shared.response import UnifiedResponse
```

业务模块内部使用绝对导入虽然稍长，但项目结构变化时更容易全局搜索和替换，也更适合学习阶段建立清晰认知。

## 迁移执行顺序

虽然本次计划一次性迁移，但执行时仍然需要有顺序，避免同时改太多以后难以定位错误。

建议顺序如下：

1. 创建目标目录：`shared`、`modules`、`modules/videos`、`modules/users`、`modules/api_tokens`。

2. 先迁移公共数据库能力，把 `Base`、`engine`、`AsyncSessionFactory`、`get_db_session` 放到 `shared/db.py`。

3. 建立模型注册入口，确保 `Video`、`User`、`ApiToken` 都能被统一导入。

4. 调整 Alembic `env.py`，让它从新的 `shared/db.py` 获取 `Base.metadata`，并导入模型注册入口。

5. 迁移 `users` 模块，因为它提供当前用户和管理员权限依赖，其他模块会依赖它。

6. 迁移 `api_tokens` 模块，保留原有 `/auth` 路由前缀，避免接口契约变化。

7. 迁移 `videos` 模块，把视频路由改为依赖 `modules/users/deps.py` 和 `modules/videos/deps.py`。

8. 创建 `modules/router.py` 聚合路由，并调整 `main.py`。

9. 全局搜索旧路径，确认 `app.api.v1`、`app.crud`、`app.service`、`app.models`、`app.schemas` 等旧导入已经清理。

10. 删除旧目录中已经迁移完成的文件，保留必要的空目录时使用 `__init__.py`，不保留两套可用代码。

## 风险清单

### 旧导入路径遗漏

迁移后最常见的问题是某个文件仍然从旧路径导入，例如：

```python
from app.schemas.video_schema import VideoCreate
from app.crud.crud_video import VideoCRUD
```

这类问题可以通过 `compileall` 和全局搜索快速发现。

### Alembic 识别不到模型

模型迁移到业务模块后，如果 Alembic 没有导入模型注册入口，`Base.metadata` 会不完整。这个问题必须在迁移当天处理，不能留到以后。

### 依赖注入重新集中

如果为了省事把所有 `get_xxx_service` 继续放在一个全局 `deps.py`，短期能跑，长期会让模块化失去意义。业务专属依赖应该放在业务模块自己的 `deps.py`。

### 路由前缀变化导致前端失效

前端当前调用的是：

```txt
/api/v1/user/login
/api/v1/videos
/api/v1/videos/create
/api/v1/videos/{id}
/api/v1/videos/condition-search
```

迁移后应该保持这些路径不变。目录变化不应该影响外部 API 契约。

### API Token 模块命名混淆

当前 `auth.py` 实际管理 API Token，而用户登录在 `user.py`。迁移时建议目录名使用 `api_tokens`，但路由可以继续保持 `/auth`。这样代码语义更准确，对外接口也不受影响。

## 完成标准

本次迁移完成时，应满足以下条件：

```txt
1. 后端不再以 api/service/crud/models/schemas 作为主要业务目录。
2. videos、users、api_tokens 三个业务模块拥有自己的 router/deps/service/crud/model/schema。
3. shared 层只保存跨业务基础设施，不包含具体业务服务。
4. main.py 通过 modules/router.py 聚合业务路由。
5. Alembic 可以从新的模型注册入口拿到完整 metadata。
6. 后端可以正常启动。
7. 前端登录、视频列表、视频新增、编辑、删除路径保持不变。
8. compileall 通过。
```

## 迁移后的预期收益

完成后，项目结构会更接近真实业务系统的长期组织方式。以后新增一个业务时，可以直接新增一个模块目录，而不是同时修改全局 `api`、`service`、`crud`、`models`、`schemas` 五个目录。

例如后续新增专辑业务时，目标会非常明确：

```txt
app/modules/albums/
  router.py
  deps.py
  service.py
  crud.py
  model.py
  schema.py
```

这种结构对小项目同样有价值，因为它会强迫开发者从一开始就思考业务边界，而不是等代码变多以后再回头整理。
